# CreatiScout AI-Native UX Design

## 1. Product Feeling

CreatiScout should feel like a **KOL digital employee workroom**, not a traditional SaaS dashboard and not a pure chat agent.

The user should feel:

- Lucy is already working in the background.
- Campaign progress is structured, visible, and deterministic.
- AI decisions are explainable through evidence and business objects.
- Humans only approve, correct, or take over high-risk moments.
- Chat exists as a command layer, not the main product surface.

Core product metaphor:

> A digital KOL operator is running the campaign lifecycle. The user supervises decisions, evidence, and exceptions.

## 2. Visual Direction

The visual style should be warm, light, and premium.

Use:

- Milk-white / cream product background.
- Soft glass-like cards.
- Pink as the primary brand color.
- Teal and pale olive as secondary signal colors.
- High information density with calm spacing.
- Fine borders, soft shadows, and rounded panels.

Avoid:

- Heavy SaaS blue-gray dashboards.
- Pure dark AI tool aesthetics.
- Generic kanban boards.
- Chat-first layouts.
- Decorative AI gradients that do not carry workflow meaning.

## 4. Core UX Model

The product should be organized around four workflow phases:

1. **Strategy Setup**
   - Brand insight / SOW judgment
   - Brief structuring
   - Media plan and budget split
   - Competitor creator strategy

2. **Creator Sourcing**
   - Creator search / targeted expansion
   - Agency / MCN resource inventory
   - AI screening + sample QA

3. **Deal & Production**
   - Outreach / email A/B testing
   - Reply screening / quote judgment
   - Contract / payment / rights
   - Samples / Coupon / One link
   - Script and video review

4. **Tracking & Learning**
   - Published data monitoring
   - Review / next strategy recommendation

Every step should expose:

- **Status**: running, completed, needs approval, blocked.
- **What Lucy did**: scan, rank, draft, compare, flag, summarize.
- **Evidence**: source data, rules, samples, benchmarks.
- **Output object**: creator list, brief, draft, contract task, content review, tracking report, strategy rule.
- **Human action**: approve, edit rule, ask why, take over.

## 5. Key Screens

### Digital Employee Workroom

Purpose: the main AI-native workspace.

Layout:

- Left sidebar: product identity, navigation, Lucy status, operating rules.
- Main area: 14-step lifecycle map grouped into 4 phases.
- Right rail: approval queue, risk queue, `@Lucy` command layer.
- Bottom area: work artifacts generated by Lucy.

This screen should communicate:

> Lucy is running the campaign while humans approve decisions.

### Campaign War Room

Purpose: a campaign-level operating map.

Layout:

- Top: campaign context, budget, spend, GMV, ROAS.
- Left: current business objects.
- Center: four-phase signal map.
- Right: Lucy operator panel with changes, next best action, evidence freshness, and command layer.

This screen should feel like a live operating room, not a project board.

### Step Evidence Drawer

Purpose: make AI decisions trustworthy.

Layout:

- Main area: step detail with input, Lucy action, evidence, output.
- Right drawer: evidence cards, human boundary, approval actions, take-over option.

Use this pattern whenever the user asks:

- Why did Lucy recommend this?
- What data did she use?
- Can I trust this shortlist?
- What happens if I approve?

### Human Intervention Moment

Purpose: show human-in-the-loop without slowing automation.

Layout:

- Main queue: high-risk approval rows.
- Right rail: Lucy recommendation, what she already handled, what needs judgment, command layer.

Approval rows should include:

- Quote exceptions.
- Contract and usage rights.
- Script and video review.
- Campaign report finalization.

## 6. Component Patterns

### Workflow Step Card

Each step card should include:

- Step number.
- Step title.
- One-line explanation.
- Status badge.
- Output object chip.

The step should be compact but auditable.

### Approval Queue Row

Each approval row should include:

- Decision title.
- Why it matters.
- Evidence tags.
- Primary action.
- Secondary action.

Recommended actions:

- Approve
- Review
- Ask why
- Edit rule
- Take over

### Evidence Card

Evidence cards should be small, specific, and source-oriented.

Examples:

- Audience fit
- Quote benchmark
- Content sample
- Brand safety
- Data freshness
- Rule applied

### @Lucy Command Layer

The command layer is a small contextual input, usually in the right rail.

Use it for:

- Asking why.
- Summarizing current state.
- Rerunning a scan.
- Updating rules.
- Turning insight into strategy memory.

Do not make chat the primary page.

## 7. Interaction Rules

- AI output must write back to a business object.
- High-risk actions require approval.
- Users can take over any step.
- Every recommendation needs evidence.
- Every rejection should become memory.
- Every approval should update the campaign state.
- Command responses should produce structured changes, not just text replies.

## 8. Tone and Copy

Use operational, confidence-building language.

Good:

- `Lucy prepared 23 outputs · 8 decisions need you`
- `Review exceptions, not every task`
- `Work artifacts, not chat bubbles`
- `AI decisions are explainable, auditable, and easy to take over`
- `Humans decide only at the high-risk edges`

Avoid:

- Generic task labels like `Todo`, `In progress`, `Done`.
- Overly magical AI copy.
- Explaining the UI inside the UI.
- Long chat-style assistant paragraphs.

## 9. Design Principle

The experience should make AI feel like a reliable operator:

> Automated enough to save work, structured enough to trust, and controllable enough to approve.
